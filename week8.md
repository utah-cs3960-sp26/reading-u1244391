The master/main branches are for the production ready and polished code. the develop branch is where the new features and branches converge and accumulate to get ready for the next release. A feature branch is one that splits off develop and merges back to develop to prepare and stage future releases but the hotfix splits from master/main and merges back into both main and develop. The difference is hotfixes address critical fixes that cant wait for the normal release cycle while feature branches are simply adding features for future releases. Release branches are branched from develop when the new features/fixes are stable for release and exist for a last minute polish. they merge onto master/main and develop when finished. If you arent going to support older version of the software the hotfix and release branches become unnecessary you just push forward changes as they are wanted. Feature branches are still useful for developing new features and isolating the work in progress. You likely dont need develop and main branches. The separation is for version releases but if you arent looking to support older versions you can just deploy changes directly to the main branch.

1. The incorrect law being tested was:
reverse (xs++ys) == reverse xs++reverse ys
QuickCheck came back with a really simple counterexample xs = [2] and ys = [-2, 1]. If you plug those into the incorrect law and just work it out step by step you can see the issue pretty clearly. On the left side you do reverse ([2] ++ [-2,1]) which combines into [2,-2,1] and then reverses to [1,-2,2]. On the right side you compute reverse [2] ++ reverse [-2,1] which gives [2] ++ [1,-2] or [2,1,-2]. Those two results [1,-2,2] and [2,1,-2] are obviously not the same so the law doesnt hold. The mistake is that the incorrect version keeps the pieces in the same order after reversing them when it actually should flip them. When you reverse a concatenation you have to reverse the order of the two lists too otherwise it just doesnt work. That is why the correct law is reverse (xs ++ ys) == reverse ys ++ reverse xs even though at first it might not seem like it.

2. The precondition ordered xs basically acts like a filter on the random lists that get generated and as the lists get longer they are way less likely to actually be ordered. A list with length 0 or 1 is always ordered so those always pass. A list of length 2 is ordered about half the time. For length 3 its more like 1 out of 6 and it just keeps getting worse from there. So the longer the random list is the more likely it just gets thrown out by the precondition and never even tested. Because of that the test cases that actually make it through are mostly really short lists. The paper says that 49% of them were empty and 32% had exactly one element which means 81% of the cases that passed had one element or less. That kind of shows how biased the testing becomes when you use a strong precondition like that.

3. For properties without any conditions QuickCheck just makes 100 random inputs and checks the property on each one and thats it. But conditional laws that use ==> work a little different. An input only counts if it actually satisfies the precondition. If it doesnt then QuickCheck just throws it away and tries again like it never happened. The whole 100 and 1000 thing is there for when the precondition is kind of rare. QuickCheck wants to find 100 valid test cases where the precondition is true. But it is not going to search forever. It will try at most 1000 total inputs. If it goes through all 1000 and still doesnt get 100 valid ones it just stops and reports however many it managed to find like saying Arguments exhausted after 64 tests. At that point its up to the programmer to decide if 64 passing tests is good enough or if they should write a custom generator so more of the inputs actually satisfy the precondition.
